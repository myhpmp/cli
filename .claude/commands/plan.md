# Planning Meeting

새 기능 추가, 구조 변경 등 기획이 필요한 작업 전에 가상 팀 기획 회의를 진행합니다.

## 언제 사용하는가

- 새 기능/커맨드 추가
- 아키텍처 또는 구조 변경
- 기존 기능의 동작 방식 변경
- 사용자에게 영향을 주는 변경

## 사용하지 않는 경우

- 버그 수정 (원인이 명확한 경우)
- 단순 리팩터링 (동작 변경 없음)
- docs/config 수정
- 질문, 조사, 디버깅

## 프로세스

**반드시 아래 순서를 따르세요.**

### 1. 요청 정리

사용자의 요청을 먼저 정리합니다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📝 요청 사항
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[사용자가 원하는 것을 1~3문장으로 요약]
```

### 2. 팀 기획 회의

3명의 팀원이 각자 관점에서 기획을 검토하고 의견을 제시합니다.

#### 참석자

| 역할 | 이름 | 검토 관점 |
|------|------|-----------|
| PM | 민수 | 요구사항 정의, 우선순위, 범위 설정, 사용자 시나리오, 엣지 케이스 |
| 시니어 개발자 | 현우 | 기술적 실현 가능성, 공수 판단, 기존 코드 영향도, 구현 방식 제안 |
| UX 디자이너 | 지윤 | CLI 사용자 경험, 출력 포맷, 에러 메시지, 명령어 네이밍 |

#### 회의 형식

각 팀원이 순서대로 발언합니다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🗣️ [역할] 이름
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[의견: 찬성/반대/대안 + 구체적 근거]
```

### 3. 토론

팀원 간 의견이 다르면 자유롭게 토론합니다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
💬 토론
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[팀원들이 서로 의견을 주고받으며 최선의 방안을 도출]
```

### 4. 결론

토론 결과를 정리합니다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 기획 결론
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

구현할 내용:
- [합의된 구현 사항 목록]

구현하지 않을 내용:
- [논의 후 제외하기로 한 사항 + 이유]

추후 과제:
- [이번에는 안 하지만 나중에 할 사항]

예상 영향 범위:
- [변경될 파일/모듈 목록]
- [DB 스키마 변경, 새 RPC 함수, 마이그레이션 필요 여부]
```

### 5. 사용자 확인

결론을 사용자에게 보여주고 **승인을 받은 후** 구현을 시작합니다. 사용자가 수정을 원하면 토론을 재개합니다.

## 주의사항

- 팀원들은 **무조건 찬성하지 않습니다**. 기술적으로 어렵거나 UX가 나쁘면 반대하고 대안을 제시합니다.
- PM은 **범위를 제한**하는 역할입니다. "이것도 추가하면 좋겠다"를 경계합니다.
- 시니어 개발자는 **기존 코드와의 정합성**을 반드시 확인합니다.
- 기획 단계에서 코드를 작성하지 마세요. 합의 후 구현합니다.
